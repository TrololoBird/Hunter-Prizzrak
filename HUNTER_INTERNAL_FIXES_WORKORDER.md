## 1. Scanner — гейт/де-ранк эмиссии на уже вычисляемом `ltf_confirmed`

**Проблема `[подтверждено кодом]`.** В `_advance_pattern_c` (`patterns.py`, ~строка 889) микро-подтверждение считается, но НЕ гейтит выпуск:
```python
setup.score = 1.0 if ltf_confirmed else 0.7      # ← только score
setup.steps_covered = _C_TOTAL_STEPS if ltf_confirmed else _C_TOTAL_STEPS - 1
```
Полный actionable-план (вход/добор/стоп/TP) выпускается при `ltf_confirmed=False` («Разворот не найден», Шаги 1/2). Реализованная изнанка — ALLO −13.55%. Аналогично проверить Pattern A/A3/B (там тоже `1.0 if ltf_confirmed else 0.7`, напр. A3 ~607, B ~769).

**Изменение.** Не выпускать неподтверждённый сетап как actionable ИЛИ визуально дизавуировать в доставке. Минимальный вариант (предпочтителен, обратимо-совместимый): оставить эмиссию, но в `deliver/manipulation_delivery.py` при `micro_confirmed=False` рендерить план как **«ожидание подтверждения — не вход»** (без чисел добора как готовых), а не как исполнимый. Радикальный вариант (если политика позволяет): гейтить эмиссию на `ltf_confirmed`. Выбор — за владельцем; по умолчанию делать минимальный.

**Критерий приёмки.** Тест: сетап с `micro_confirmed=False` не рендерится как исполнимый план (нет «📍 Вход (рыночный/лимит)» как actionable); с `True` — рендерится. Существующие тесты доставки зелёные.

**Риск.** Низкий; трогает только представление/гейт, не детекцию.

## 2. Scanner — портировать reachable-target Pattern B → Pattern C (fantasy-RR)

**Проблема `[подтверждено кодом]`.** Pattern C берёт цель из дальнего пула: `target = ladder[0]` (`patterns.py` ~881) из `_target_ladder`, который ранжирует пулы **по дистанции — «дальше = лучше»** (~306: `score = pct * (1.0 + touches…)`). Итог — TP1 на +46…+88% (CRV: R:R 8.36 на +88%). Pattern B **уже решил ровно это**: см. комментарии `patterns.py` ~628–748 — «deepest-pool target manufactured 57 timeouts», «ZEC #18718: TP 220.5, −57%, RR 14.43», B переключён на достижимую цель (impulse/структурный low), НЕ на дальний пул.

**Изменение.** В блоке C заменить `ladder[0]` на достижимую цель по образцу B (impulse-set экстремум / ближний структурный уровень). Дистанцию цели **скейлить по мезо-TF лестницы** (1h-лестница → ближе, 1d → дальше) — не единый «ближайший». Более дальние пулы можно оставить как справочную лестницу, но `primary_target`/TP1 должен быть достижимым.

**Критерий приёмки.** Пиннинг-тест: на CRV-подобном входе TP1 не проходит как +88% (например, R:R до TP1 ограничен разумным потолком по мезо-TF); тест на скейл дистанции по TF. Регресс сканера зелёный.

**Риск.** Средний (меняет R:R-фильтр и таргеты). Обязателен пиннинг на нескольких символах.

## 3. Scanner — геометрия входа: не заходить ниже реклейма; сузить полосу

**Проблема `[наблюдение из вывода — верифицировать в коде].`** В живых сигналах нижний край полосы входа уходит **ниже** свипнутого уровня: CRV entry `0.20505` < swept `0.20890`; HEI `0.10244` < swept `0.10494`; Д1 внутри свипнутой территории. Вход/добор ниже реклейма = покупка в продолжение свипа, что противоречит «reclaim». Плюс полоса входа бывает очень широкой (BILL 12.6%: `0.05272–0.05937`) — «зона» теряет смысл, исход определяется филом.

**Верифицировать.** Найти, где строится полоса входа для C (`_build_setup` / `_geometry` / `entry_ref`-логика в `patterns.py`), и проверить: (а) может ли нижний край опускаться ниже `swept_level`/реклейм-уровня; (б) чем ограничена ширина.

**Изменение.** (а) Нижний край входа не ниже реклейм-уровня (или явно помечать доборы ниже реклейма как «против реклейма, повышенный риск»). (б) Кап ширины полосы (например, ≤ k·ATR% мезо-TF) или явная пометка «широкая зона — R:R зависит от фила».

**Критерий приёмки.** Тесты: полоса входа C не опускается ниже swept-уровня (или помечена); ширина ≤ порога/помечена. R:R по-прежнему считается от worst-edge (сохранить — это корректно).

**Риск.** Средний; менять аккуратно, геометрия влияет на стоп/добор.

## 4. Scanner — «почему»/score различающими или убрать

**Проблема `[подтверждено кодом]`.** `evidence` в C (`patterns.py` ~872–874) — фиксированный список из предусловий:
```python
evidence = ["prior_swing_high", "zakrep_reclaim", "wide_stop_dobor",
            "ascending_channel_pre" if asc_ok else "",
            "descending_channel_post" if desc_ok else ""]
```
Но `asc_ok`/`desc_ok`/`vol_ok`/`zakrep` — жёсткие гейты выше (`if not …: return`), поэтому **каждый** выпущенный C имеет все токены всегда → «почему» не различает сигналы (BILL=CRV=HEI дословно). Score 0.7 при этом осмыслен (pre-confirmation), но «почему» — тавтология.

**Изменение.** «Почему» показывать только **варьирующиеся** факторы (напр. `htf_bullish`/`htf_bearish`, `micro_confirmed`, глубина закрепа `zakrep`-число, число касаний, R:R-класс), а предусловия не выводить как «причины». Либо убрать строку «почему», если варьирующегося нет.

**Критерий приёмки.** Тест: два C-сетапа с разными htf_bias/закрепом/подтверждением дают **разные** строки «почему». Идентичные предусловия не порождают идентичную тавтологию.

**Риск.** Низкий (представление).

## 5. Prizrak — проверить немонотонность HTF-весов

**Проблема `[подтверждено кодом — требует решения владельца].`** `prizrak/config.py` ~89–92:
```python
htf_1w_weight = 0.35   # 1w
htf_1d_weight = 0.25   # 1d
htf_4h_weight = 0.30   # 4h  ← выше, чем 1d
htf_1h_weight = 0.10   # 1h
```
Веса **немонотонны по ТФ**: `4h (0.30) > 1d (0.25)` — старший таймфрейм весит меньше младшего. Это напрямую влияет на HTF-bias-балл (тот самый `ШОРТ (−0.60)` в BTC-сигнале). Может быть намеренно (4h реактивнее), может — опечатка/недосмотр.

**Изменение.** НЕ менять вслепую. Задача: (а) найти обоснование в истории/докстрингах; (б) если намеренно — задокументировать почему `4h>1d` одной строкой в комментарии; (в) если нет — сделать монотонным (`1w ≥ 1d ≥ 4h ≥ 1h`) и пере-пиннить сигналы, где bias меняет знак.

**Критерий приёмки.** Либо явный комментарий-обоснование немонотонности, либо монотонные веса + обновлённые пиннинг-тесты HTF-bias. Не оставлять без решения.

**Риск.** Средний (меняет направление bias). Требует пиннинга.

## 6. Maps/delivery — доставка глубоких стен + честный venue-лейбл

**Проблема `[наблюдение из вывода + частично код].`**
- **Глубокие стены детектятся, но не доставляются.** `_detect_sticky_walls` (`maps/orderbook.py` ~124) трекает `notional_usd`+`persists`+`distance_pct` (топ-6 по дистанции), но DOM-секция в `/signal` показывает только near-price полосу → стены на 1.5–3% от цены (напр. 61000/61750 по внешним данным) не доходят до текста.
- **`venue_completeness` не различает «венью жив, 0 событий» и «фидер мёртв».** Ликв-лейбл показывает только `binance=capped_1s`; неясно, жив ли bybit-realized-фидер. `liq_synthetic_only` (`maps/liquidation.py` ~98) = `realized_event_count==0`.

**Изменение.**
- В доставке DOM/стен (`deliver/_sections.py`) рендерить топ-N устойчивых стен по нотионалу в радиусе ±3–4% от цены, а не только верх стакана (данные уже в `sticky_walls`, 0 новой сети).
- В ликв-лейбле перечислять **все живые венью** с числом событий (`bybit=live/0ev`, `binance=capped_1s`, …), чтобы отличать тихий рынок от упавшего фидера.

**Критерий приёмки.** Тесты: (а) стена на −2% от цены попадает в доставку; (б) лейбл различает live-0ev и отсутствующий венью. Регресс доставки зелёный.

**Риск.** Низкий (представление/лейблы, данные уже есть).

## 7. Что НЕ строить (результат necessity-review — не реализовывать)

Эти пункты **провалили тест необходимости**; не тратить на них время без нового доказательства нужды:

- **Опционный слой (GEX/max-pain/IV, Deribit)** — ❌ **universe-mismatch.** Deribit-опционы существуют только для BTC/ETH/AVAX/TRX (глубина — BTC/ETH); система сигналит на низкокапные альты (BILL/CRV/HEI/ALLO/ONDO/AEVO/POL) — **опционного рынка у них нет**. Применимо к ~2 символам, только macro-тир, только у экспирации. Не строить как общий слой.
- **Футпринт/CVD как новый слой подтверждения сканера** — ❌ **избыточно.** Сканер уже считает `ltf_confirmed` (bos/choch); необходимость закрывается фиксом #1 (гейт), не новым слоем. Пересмотреть только если после #1 bos/choch окажется недостаточным — с доказательством.
- **Проводка liquidation-карты в таргеты сканера** — ❌ **избыточно.** Достижимую цель уже даёт impulse-low Pattern B; необходимость закрывается фиксом #2 (порт B→C), без нового data-wire.
- **Спот-книга, OFI/microprice/VPIN, on-chain (SOPR/LTH)** — ❌ **недоказанная необходимость** и/или чужой горизонт/устав. Не строить.

Причина отказа общая: **не добавлять data-слой, пока внутренняя корректность не исчерпана и необходимость слоя не доказана фактом** (та же дисциплина, что evidence-gate 1в-2).
